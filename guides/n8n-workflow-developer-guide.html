Build Prosponsive-Compatible Tools with n8n

n8n Workflow Developer Guide

This guide is for technical workflow developers who want to build n8n workflows that work as tools inside Prosponsive. It covers the architecture, the async execution model, the API surface, tool discovery mechanics, and a step-by-step tutorial for building your first tool.

By the end of this guide you will understand:

  • How Prosponsive discovers and executes n8n workflows as agent tools
  • The full asynchronous round-trip between Prosponsive and n8n
  • The API endpoints your workflows interact with
  • How to structure workflows so Prosponsive can find, describe, and invoke them
  • Troubleshooting strategies for common integration issues

1. Architecture Overview

Prosponsive is an Electron desktop application that runs a local HTTPS API server. n8n runs in a container (Podman or Docker) alongside PostgreSQL on the same machine. All communication between these components uses HTTPS with locally-generated SSL certificates.

Component layout

ComponentWhere it runsPort
ProsponsiveElectron main process (host OS)HTTPS on port 3100
n8nContainer (Podman or Docker)HTTPS on port 5678
PostgreSQLContainer (Podman or Docker)Port 5432

Network topology

Because n8n runs inside a container, it cannot reach the host machine at localhost or 127.0.0.1. Prosponsive uses a single, consistent callback hostname regardless of container runtime: host.containers.internal.

Podman provides this hostname natively. For Docker, Prosponsive's compose file adds an extra_hosts entry that maps host.containers.internal to Docker's host-gateway, so the same hostname resolves in both environments.

The full callback URL is always https://host.containers.internal:3100. This is the address n8n uses to POST results back to Prosponsive.

HTTPS is required. Callback URLs must use HTTPS. The external hook silently refuses to send results to HTTP callback URLs -- there is no error message, the callback simply never arrives. If tool executions complete in n8n but results never appear in Prosponsive, verify that the callback URL uses https://, not http://.

2. How Tool Discovery Works

Prosponsive automatically discovers n8n workflows and presents them as tools available to agents. Understanding this process is essential for building workflows that Prosponsive will recognize.

Discovery process

Discovery happens in two phases: an initial filter pass over the workflow list, followed by a process pass that fetches each workflow's details.

Phase 1 -- Filter (operates on the workflow list summary):

  1. Polling -- The Tool Discovery Service polls the n8n API every 15 minutes to fetch all workflows.
  2. Active check -- Only workflows that are active (published) pass the filter. Inactive workflows are invisible to Prosponsive.

Phase 2 -- Process (operates on each surviving workflow's full detail):

  1. Adapter exclusion -- The adapter workflow itself (Prosponsive: n8n Workflow Adapter) is skipped.
  2. Webhook detection -- The workflow must have a Webhook trigger node with a path parameter. Workflows without a webhook trigger are skipped. (Note: this check happens during processing, not during the initial filter.)
  3. Name derivation -- The workflow display name is converted to snake_case to produce the tool name. For example, Email Move To Folder becomes email_move_to_folder.
  4. Description extraction -- The tool description is sourced in priority order: (a) meta.description on the workflow, (b) the first paragraph of a Sticky Note node, (c) an auto-generated description from the workflow name.
  5. Parameter extraction -- Code nodes are scanned for patterns like body.param_name || default and parameters.param_name ?? default to extract input parameters, types, and defaults.
  6. Caching -- Discovered tools are cached with a 5-minute TTL. Stale entries are pruned on each refresh cycle.

Tool naming rules

The tool name is derived from the workflow display name:

  1. Convert to lowercase
  2. Remove all characters except letters, numbers, and spaces
  3. Trim whitespace
  4. Replace spaces with underscores

3. The Async Execution Model

Tool execution in Prosponsive is fully asynchronous. When an AI agent proposes a tool call, the execution follows a multi-step round-trip through the adapter workflow. This section explains every step.

The Prosponsive Tool Adapter

All tool executions are routed through a single adapter workflow called Prosponsive: n8n Workflow Adapter. This adapter is a custom n8n node (ProsponsiveToolAdapterTrigger) that:

  1. Receives the tool call request from Prosponsive
  2. Looks up the target workflow by workflowId
  3. Validates the target workflow is active and has a supported trigger node
  4. Executes the target workflow, injecting parameters as pin data
  5. Returns 202 Accepted immediately so Prosponsive is not blocked
  6. After the target workflow completes, an external hook sends results back to Prosponsive via the callback URL

Individual tool workflows do not need to handle callbacks, authentication, or routing. The adapter handles all of that plumbing.

Advanced note: While tool discovery requires a Webhook trigger node (so Prosponsive can determine the webhook path), the adapter itself can route to workflows with five trigger types: webhook, executeWorkflowTrigger, chatTrigger, formTrigger, and scheduleTrigger. This means you can build adapter-invocable workflows that use non-webhook triggers, but they must be registered through a different mechanism (such as a wrapper workflow with a webhook trigger that calls them via the Execute Workflow node).

4. Prosponsive API Endpoints

These are the HTTP endpoints that Prosponsive exposes on its local API server (port 3100). While most of the callback plumbing is handled automatically by the adapter and external hooks, understanding these endpoints is valuable for debugging and building advanced workflows.

Sends a prompt to the active AI provider with full conversation history. Used by n8n workflows that need to invoke AI reasoning as part of their processing.

  "stopReason": "end_turn",

Same as /api/ai/inference but does not include conversation history in the provider request. Takes an explicit agentId and uses that agent's system prompt and tool definitions. Useful for stateless, one-shot AI calls from within workflows.

4.6 POST /api/conversations/resolve

Resolves a conversation by name, creating it if it doesn't exist. This is how workflows route messages to a specific conversation in the Prosponsive UI. The imap-new-email workflow uses this to ensure all inbound emails land in the "Email Inbox" conversation.

Request body:

{
  "slug": "Email Inbox"
}
FieldTypeRequiredDescription
slugstringYesThe conversation name (e.g., "Email Inbox", "Support Tickets"). Prosponsive derives a URL-safe slug from this value (e.g., email-inbox) and uses the original value as the display name. Must not be empty or include the slug: prefix.

Response body:

{
  "success": true,
  "conversationId": "a1b2c3d4-...",
  "slug": "Email Inbox"
}

Behavior:

  • Idempotent: If a conversation with the given name already exists, its conversationId is returned. Calling this endpoint repeatedly with the same name is safe and returns the same conversation.
  • Auto-creates: If no matching conversation exists, one is created automatically and the renderer is notified so the sidebar refreshes immediately.
  • The value is stored as both the conversation's slug (for lookup) and its externalId (for external system correlation).

Typical usage pattern: Call this endpoint first to get a conversationId, then pass it to /api/ai/agent-inference so the AI response appears in the correct conversation. For example:

// Step 1: Resolve conversation by name
POST /api/conversations/resolve
{ "slug": "Email Inbox" }
// Returns: { "conversationId": "a1b2c3d4-..." }

// Step 2: Send AI inference into that conversation
POST /api/ai/agent-inference
{
  "prompt": "New email from alice@example.com...",
  "conversationId": "a1b2c3d4-...",
  "agentId": "executive-assistant-email-agent"
}
Tip: Use a consistent, descriptive name for each workflow's target conversation. All executions of a workflow that resolve the same name will feed into the same conversation, giving the user a single place to see all related activity.

5. How Parameters Are Extracted

Prosponsive reads your workflow's code nodes to discover tool parameters. This is how the AI knows what inputs your tool accepts. Understanding the extraction patterns lets you design workflows that advertise the right parameters.

Pattern 1: Parameters with defaults

Code that reads from the request body with a fallback value:

const limit = body.limit || 10;
const format = body.format ?? 'json';

Produces:

ParameterTypeRequiredDefault
limitnumberNo10
formatstringNo"json"

Pattern 2: Required parameters (no default)

Code that reads from the request body without a fallback:

const uid = body.uid;
const mailbox = body.mailbox;

Produces:

ParameterTypeRequired
uidstringYes
mailboxstringYes

Important: The required-parameter scanner only matches const declarations (e.g., const uid = body.uid). Variables declared with let or var will not be detected as parameters. Always use const when reading from body.* so that Prosponsive discovers them correctly.

Pattern 3: Nested parameters object

Code that reads from parameters.* (the adapter wraps inputs in a parameters envelope):

const query = parameters.query ?? '';
const maxResults = parameters.maxResults ?? 25;

Produces:

ParameterTypeRequiredDefault
querystringNo""
maxResultsnumberNo25

Type inference

The type is inferred from the default value:

Default value patternInferred type
Integer (e.g., 10, 0)number
true or falseboolean
Starts with [array
Starts with {object
Anything elsestring
Best practice: Use explicit default values in your code nodes so Prosponsive can infer the correct types. A code node with const limit = parameters.limit ?? 10 advertises a numeric parameter. Without the default, it defaults to string.

Request Context and tool parameter mapping

When a workflow sends data to Prosponsive (e.g., the imap-new-email workflow posting an email notification), that data arrives as Request Context — a JSON object attached to the message. You can see it on any message that carries it by clicking the Request Context expander in the conversation.

Request Context fields become available for tool parameter mapping in agent definitions. This is how you connect incoming data to outgoing tool calls — for example, mapping the uid field from an incoming email notification to the uid parameter of the email_move_to_folder tool.

How Request Context flows

  1. Workflow sends data — Your n8n workflow calls /api/ai/agent-inference with a context object. This becomes the message's Request Context.
  2. Prosponsive learns the fields — As messages arrive, Prosponsive samples Request Context payloads and discovers all available field paths (including nested fields up to 4 levels deep).
  3. Fields appear in the agent editor — The discovered fields populate the Request Context field picker in the Tool Parameter Guidance Builder.
  4. Agent uses mapped values — When the agent calls a tool, the generated guidance in its system prompt tells it to read parameter values from the Request Context instead of asking the user.

Request Context is cumulative

The set of available Request Context fields is a superset of all fields seen across the conversation, not just the fields from a single message. Prosponsive samples every message that carries Request Context and merges their field paths together. This means different workflows — or different executions of the same workflow — can contribute different fields, and all of them become available for parameter mapping.

For example, imagine a conversation that receives two types of messages:

  • An inbound email notification with fields: uid, mailbox, from, subject, body
  • A calendar event reminder with fields: eventId, title, startTime, attendees

The Tool Parameter Guidance Builder would show all nine fields combined: uid, mailbox, from, subject, body, eventId, title, startTime, attendees. When mapping parameters for a tool, you pick from the full set — Prosponsive handles matching the right field to the right message at runtime.

Mapping parameters in the agent editor

Each agent has a Tool Parameter Guidance Builder in its editor sidebar. To map a tool's parameters:

  1. Open the Agents tab and select your agent.
  2. Scroll down to the Tool Parameter Guidance Builder at the bottom of the editor.
  3. Select a tool from the dropdown — its discovered parameters appear.
  4. For each parameter, choose a source:
    • Request Context field — Use a value from the trigger data. A field picker shows all discovered paths (e.g., uid, mailbox, from). Best for automated workflows where the data is already present.
    • Constant value — Always use a fixed value. Best for parameters that never change (e.g., a default mailbox name).
    • Ask at runtime — The agent asks the user before calling the tool. Best for interactive conversations.
  5. The builder generates a natural language guidance block. Copy it and paste it into your agent's definition (in the markdown body). This becomes part of the agent's system prompt.

Example: email agent parameter mapping

The imap-new-email workflow sends Request Context containing email fields like uid, mailbox, from, subject, and body. The email agent's definition maps these to the email_move_to_folder tool:

## Tool Parameter Guidance
When using the **email_move_to_folder** tool:
Purpose: Move an email to a different IMAP folder
Parameter sources:
- **uid**: Use the "uid" value from the Request Context. If missing, ask the user.
- **mailbox**: Use the "mailbox" value from the Request Context. If missing, ask the user.
- **destination**: Ask the user for this value before calling the tool.

This tells the agent: "When you need to move an email, get the uid and mailbox from the message data automatically, but ask the user which folder to move it to."

Tip: Request Context field discovery is based on sampling — fields only appear in the picker after at least one message carrying those fields has been received. To see what fields are available, find a message in the conversation that represents the kind of trigger your tool should act on (e.g., an inbound email notification) and expand its Request Context. Those are the fields you can map to tool parameters in the agent definition.

6. Build Your First Tool

Let's walk through creating a simple tool workflow that Prosponsive can discover and invoke. This example creates a "Hello World" tool that accepts a name parameter and returns a greeting.

Step 1: Create the workflow

  1. Open n8n (click the n8n tab in Prosponsive, then Workflows).
  2. Click Add Workflow (or the + button).
  3. Name the workflow: Hello World Greeter

The tool name will be derived as hello_world_greeter.

Step 2: Add a Webhook trigger

  1. Click Add first step and search for Webhook.
  2. Select the Webhook node.
  3. Set the HTTP Method to POST.
  4. Set the Path to tool-hello-world-greeter.
  5. Leave Respond set to Immediately (the adapter handles response timing).

Step 3: Add a Code node for processing

  1. Add a Code node after the Webhook trigger.
  2. Set the language to JavaScript.
  3. Enter the following code: 
    // Extract parameters from the adapter envelope
const body = $input.first().json.body || $input.first().json;
const parameters = body.parameters || body;

// Read inputs -- the patterns below are what Prosponsive scans for discovery
const name = parameters.name ?? 'World';
const enthusiasm = parameters.enthusiasm ?? false;

// Build greeting
const greeting = enthusiasm
  ? `Hello, ${name}! Great to meet you!`
  : `Hello, ${name}.`;

return [{
  json: {
    greeting,
    generatedAt: new Date().toISOString()
  }
}];

This code will cause Prosponsive to discover two parameters:

ParameterTypeRequiredDefault
namestringNo"World"
enthusiasmbooleanNofalse

Step 4: Add a description

Add a Sticky Note node anywhere on the canvas and type:

Generates a personalized greeting. Use this tool when the user asks to be greeted or wants a hello message.

Alternatively, set the workflow's description in Settings (gear icon in n8n) > Description field. The meta.description field takes priority over sticky notes.

Step 5: Save and publish

  1. Click Save in the top-right corner.
  2. Toggle the workflow to Published (also in the top-right corner).

Step 6: Wait for discovery

Prosponsive discovers new tools on its 15-minute polling cycle. To speed things up, you can restart Prosponsive -- discovery runs on startup.

After discovery, the tool hello_world_greeter will appear in agent tool lists.

Step 7: Test from a conversation

  1. Open Prosponsive and start a conversation.
  2. Make sure your agent has access to all tools (or add hello_world_greeter to its allowedTools).
  3. Type a message like:

    Can you greet me? My name is Alex.

  4. The AI agent should propose a hello_world_greeter tool call with name: "Alex".
  5. Approve the tool card.
  6. The result appears in the conversation.

7. Agentic Loop with Multiple Tool Calls

Some conversations involve the AI calling multiple tools in sequence. This is the "agentic loop" -- the AI reasons, calls a tool, gets the result, reasons again, and may call another tool.

sequenceDiagram
    participant U as User
    participant P as Prosponsive
    participant AI as AI Provider
    participant N as n8n (via Adapter)

    U->>P: "Find my unread Project Zen emails
and move them to the Zen channel"

    P->>AI: Inference (with tool definitions)
    AI-->>P: tool_call: search_emails(query="Project Zen")
    P->>U: Show approval card
    U->>P: Approve

    P->>N: Execute search_emails
    N-->>P: Callback: [3 emails found]

    P->>AI: Inference (tool result + history)
    AI-->>P: tool_call: move_thread(target="Project Zen")
x3 (one per email)
    P->>U: Show 3 approval cards
    U->>P: Approve All

    P->>N: Execute move_thread (x3)
    N-->>P: Callbacks: [moved, moved, moved]

    P->>AI: Inference (results + history)
    AI-->>P: "Done! I moved 3 Project Zen emails
to the Project Zen channel."
    P->>U: Display final response

Each tool call in the loop goes through the same async pattern: Prosponsive calls the adapter, the adapter executes the target workflow, and the external hook sends results back via the callback endpoint.

8. Workflow Best Practices

Naming conventions

  • Use descriptive, action-oriented names: Email Move To Folder, Search Jira Issues, Get Channel Activity
  • The name becomes the tool identity -- changing a workflow name changes its tool name, which may break agent configurations that reference it

Descriptions

  • Always provide a description. Without one, the AI receives a generic auto-generated description that may not accurately convey when to use the tool.
  • Write descriptions from the AI's perspective: "Use this tool when you need to move an email to a different IMAP folder."
  • The meta.description field (workflow settings) takes priority over sticky notes.

Parameter design

  • Use the parameters.* pattern in code nodes for clear parameter extraction
  • Provide default values to make parameters optional and to help Prosponsive infer types
  • Use descriptive parameter names -- they are converted to human-readable descriptions automatically (maxResults becomes "The max results")

Output structure

  • Return data as JSON objects in the json property of output items
  • The last node's output is what gets sent back to Prosponsive via the callback
  • Keep outputs focused -- the AI will summarize the result for the user

Structured key-value display

Prosponsive automatically detects short key-value pairs in AI responses and renders them as a compact two-column grid with bold keys. This produces a polished, scannable layout instead of a plain text list.

How it works: When the AI's response contains two or more consecutive lines that match a key-value pattern, Prosponsive formats them as a two-column table -- two pairs per row, keys in bold.

Supported delimiters:

  • Key: Value (colon followed by space)
  • Key = Value (equals sign)
  • Key - Value (space-dash-space)

Rules for detection:

  • The key must be under 30 characters and start with a letter
  • There must be at least 2 consecutive key-value lines (blank lines between them are OK)
  • A bare key with no value (e.g., Body:) ends the key-value section -- everything after is treated as body content
  • URLs after a colon are not treated as key-value pairs (they become clickable links instead)

How to take advantage of this in your tool workflows: Structure your workflow's output JSON with flat, descriptive field names. When the AI summarizes the result, it will naturally present the fields as key-value lines, triggering the compact display.

For example, a tool that returns:

return [{
  json: {
    "Subject": "Re: Q3 Planning",
    "From": "alice@example.com",
    "Date": "April 3, 2026",
    "Status": "Unread",
    "Folder": "INBOX",
    "Priority": "Normal"
  }
}];

...will cause the AI to respond with lines like:

Subject: Re: Q3 Planning
From: alice@example.com
Date: April 3, 2026
Status: Unread
Folder: INBOX
Priority: Normal

Prosponsive renders this as a compact grid:

Subject:Re: Q3 PlanningFrom:alice@example.com
Date:April 3, 2026Status:Unread
Folder:INBOXPriority:Normal
Tip: Keep field names short and human-readable (e.g., "From" not "sender_email_address"). The AI will echo them as-is, and short keys produce the best two-column layout. If you have more than 8-10 fields, consider grouping them or omitting less important ones -- the compact display works best with focused, scannable data.

Error handling

  • If your workflow encounters an error, the external hook captures it and reports success: false with the error message
  • For expected failures (e.g., "email not found"), return an explicit error object: 
    return [{
  json: {
    error: "Email with UID 456 not found in INBOX",
    success: false
  }
}];

Prosponsive authenticates to n8n using an API token stored in secure storage. This is the same token you created during Prosponsive setup (see the User Guide, Section 2). The token is sent as a Bearer token in the Authorization header on webhook calls.

Request limits

  • Maximum request body size: 1 MB
  • The adapter validates that workflowId is present and the target workflow is active before executing

SymptomCauseFix
Tool executes but result never appearsSSL certificate not trusted by n8n containerEnsure the Prosponsive-generated SSL cert is mounted into the n8n container

Debugging workflow execution

  1. n8n execution log -- Open n8n and check the execution list for your workflow. Look for errors in individual node runs.
  2. Prosponsive logs -- Check ~/Library/Application Support/prosponsive/logs/combined.log for [N8N-BRIDGE] and [TOOL-DISCOVERY] entries.
  3. Health check endpoint -- POST to /api/ai/agentic-loop-health with the executionId and workflowId to confirm whether the callback was received.
  4. Adapter 400/404 errors -- If the adapter returns 400, the workflowId is missing. If 404, the workflow doesn't exist. If the workflow exists but isn't active, you get a 400 with a message about the workflow not being active.
